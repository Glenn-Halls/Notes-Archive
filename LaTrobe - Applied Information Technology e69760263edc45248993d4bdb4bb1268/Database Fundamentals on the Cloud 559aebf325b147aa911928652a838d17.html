  • Data Model Basic Building Blocks:
    • Entities:
      • A real-world object - person, place or thing - about which data will be stored
      • Rectangle in an ERD, also known as an entity box
      • Name will be singular and in all-caps
    • Attributes:
      • Characteristics about an entity
    • Relationships:
      • Associations between entities: 1:M, M:N, 1:1
      • Relation is referred to as connectivity, in the form of an active or passive verb
    • Constraints:
      • Restrictions placed on data to ensure data integrity in the form of rules
    • Business Rules:
      • Policies, procedures or principles that define the above building blocks
      • As a general rule, a noun in a business rule will translate to an entity in the model and a verb in a business rule will translate to a relationship among entities
  • Naming conventions:
    • Attributes are often in all-caps
    • It is good practice to prefix the name of an attribute with the name or an abbreviation of the name of the entity in which is occurs for example a customer credit limit = CUS_CREDIT_LIMIT
  • Schema refers to the conceptual organisation of the entire database as viewed by the database administrator
  • Subschema refers to the portion of the database ‘seen’ by the application programs
  • Data manipulation languages (DMLs) and Data Definition Languages (DDLs)
  • The relational model was introduced in 1970
  • Peter Chen first introduced the Entity Relationship Model (ERM and thus ERDs) in 1976
  • Each row in a relational table is known as an entity instance or entity occurrence with a collection of entities known as an entity set
  • The Object Orientated Model - OODM
    • both data and its relationships are contained in a single structure, known as an object
    • Object Orientated Database Management System = OODBMS
    • OODM is said to be a semantic data model
    • OODM objects contain both their attributes and also their functions / operations
      • objects are therefore self-contained and re-usable
      • objects are analogous to an ER model’s entities
      • Objects with similar characteristics are grouped in classes, which may have methods
      • Classes are organised in a class hierarchy
      • Classes have inheritance
    • OODMs are typically depicted using Unified Modelling Language (UML) class diagrams
  • The Extended Relational Database Model - ERDM
    • adds some features of the OODM within the inherently simpler RDM framework
    • includes objects, extensible data types based on class and inheritance
    • is often described as an Object / Relational Database Management System (O/R DBMS)
    • is more focused toward general deployment as opposed to OO’s specificity
    • is easily extended to accommodate new classes and capabilities such as XML
  • Extensible Markup Language (XML) has emerged as the de facto standard within the industry, in order to be able to handle the growing quantity of unstructured data such as diagrams, media, webpages etc.
  • Big data refers to the movement to find new and better ways to manage the large amounts of web and sensor generated data to derive business insights and is defined by the 3 Vs:
    • Volume
    • Velocity
    • Variety
  • Hadoop: java based, open source, high speed, distributed storage and computation. Supported by the Apache Software Foundation.
    • Hadoop Distributed File System (HDFS): storage for big data at high speeds
    • MapReduce: open source API providing fast data analytic services by distributing data processing among thousands of parallel nodes
      • Map: takes a job and splits it into smaller units of work
      • Reduce: collects outputs and recombines into a whole
  • NoSQL: large-scale distributed database system that stores both structured and unstructured data
    • Key-Value data model: ID, Key (attribute), Value (usually a string)
    • very useful for large amounts of sparse data
      • consider a clinic with many patients and many assessments - do not want a blank field for every test that a patient has NOT taken…
    • highly scalable, available and fault tolerant
  • Data abstraction:
    • External Model:
      • End User’s View of the data environment
      • External Model’s View / External Schema
    • Conceptual Model:
      • Global view of the entire database by the entire organisation
      • Conceptual Schema
      • The most widely used being the ER model represented by the ERD
      • independent from both software and hardware
      • Logical Design: creating a conceptual model that could be implemented by any DBMS
    • Internal Model:
      • Maps the conceptual model on the DBMS
      • Internal schema
    • Physical Model:
      • Lowest level of abstraction, describes the way in which data is stored and accessed in the physical world
      • software and hardware dependant
      • When you can change the physical model without affecting the internal model you are said to have physical independence

Week 2: Relational Model

  • The relational model has three well defined components:
    1. A logical data structure represented by relations
    1. A set of integrity rules to enforce that data is consistent and remains so over time
    1. A set of operations that defines how data is manipulated
  • Table: a collection of entity occurrences representing an entity set
    • tuple, attribute, attribute domain, order of rows and columns is immaterial, each row must be unique
  • Keys
    • Functional dependence where stating one value will reveal another - a key is the determinant and the information revealed is a dependant. Full functional dependence is where there are no non-necessary determinants in the determinant - dependant relationship.
    • A key is an attribute that can determine the values of other attributes
    • Primary Keys
      • One or more attributes that UNIQUELY identifies any tuple
      • May use a created dedicated ID attribute
      • Cannot be NULL, must be unique
      • In some cases, a flag / dummy code may be used as an ID eg student number -99 may represent a dummy or NULL student
    • Composite Keys
      • A primary key that has more than one key attribute
    • Foreign Keys
      • A primary key copied in another relation for cross-referencing
      • A primary key from table A that is used in table B
    • Super Keys
      • A key that can uniquely identify any row in the table
    • Candidate Keys
      • A minimal superkey - a superkey without any unnecessary attributes
    • Secondary Keys
      • Used strictly for data retrieval purposes and is an alternative to the primary key. Imagine where you have forgotten your customer number and use your name and phone number instead. May not be unique in the way a primary key is.
  • Integrity Restraints
    • Entity Integrity: the condition in which each entity instance is unique
      • Problems with an entity within a relation
    • Referential Integrity: the condition where every reference to an entity instance by another entity instance is valid
      • Problems with a reference between relations
  • Relational Algebra
    • Select / restrict (tuples)
      • σ
      • σ year ≥ 2000
      • unary operator - only one table as input, yields resulting tupules that fit the specifications without truncating attributes
    • Project (attributes)
      • π
      • π title = Hook
      • like a cut and paste for attribute columns - use 2 or more to relate to keys
      • π cus_name, cus_number (customer)
    • Union
      • ∪
      • combines all rows from two tables, excluding duplicates
      • π supplier name (supplier) ∪ π vendor name (vendor)
    • Intersect
      • ∩
      • yields only rows that appear in both tables
      • π supplier name (supplier) ∩ π vendor name (vendor)
    • Difference
      • -
      • yields the values in one table that do not appear in the other
      • NB: order is very important here
      • π supplier name (supplier) - π vendor name (vendor)
    • Product / Cartesian Product
      • x
      • yields all possible combinations of two tables
    • Join
      • ⨝
      • allows information to be intelligently combined from 2 or more tables such as when one table contains a foreign key
      • Natural join process:
        1. product of tables
        1. select the join columns
        1. project to eliminate duplicate join columns
      • customer ⨝ agent
      • equijoin
        • =
        • compares specific columns of each table, does not eliminate duplicate columns and the condition / joining criterion must be specified
      • theta join
        • theta in subscript after ⨝
        • a type of outer join
        • all other joins omit unmatched values where a theta join returns all values, with unmatched pairs being NULL
        • Depending on the filled vs the null side, a theta join can be said to be a left theta join (left filled, right null) or a right theta join (right filled, left null)
      • Divide
        • ÷
        • Determines if one set of data is associated with all values of another set of data
        • Operand is a 2 column table
        • Divisor is a 1 column table (column B)
        • yields results from column A that have every instance of column B

  • Data dictionaries
    • “The database designer’s database”
    • Includes the system catalog, can sometimes be used interchangably
    • Contain metadata about all database characteristics
    • A table describing the data types of each attribute, required or not, if it is a key or not
  • Relationships
    • one-to-one
    • one-to-many
      • most common
    • many-to-many
      • implemented as 2x one-to-many relationships
  • relvar - a relation variable ie. the variable that holds the data of the relation
    • heading: contains the names of the attributes
    • body: contains the relation
    • convention is that a relation is referred to with lowercase r and relvar with uppercase R
  • Many : Many (M:N) relationships
    • In a database scenario, these should always be broken down into two 1:M relationships to avoid redundancy issues
  • Composite / Bridge / Associative Entities:
    • Contains, at the bare minimum, the two foreign primary keys of the associated entities
      • these two foreign keys may combine to the primary key of the table
    • May be referred to as a linking table

Week 3: Database design using ER/EER diagrams

  • One to one or many, many to one or many
  • Cardinality = optionality
  • Relationship strength:
    • Strong / identifying relationships have shared attributes in the primary key and have a solid line in crow’s foot notation. These relationships are existent-dependent
    • Weak / non-identifying relationships do not share attributes in their primary key and have a dotted line in crow’s foot notation. These relationships are existent-independent
  • Required and optional attributes: required attributes are represented by bold font where optional are represented by standard font
  • Primary key identifiers are represented with an underline - this should ideally be one attribute but may be two or more in the case of a composite primary key
  • Shorthand for crows foot notation is with the following syntax:
    • CLASS (CLS_PRIMARYKEY, CLS_OTHERATTRIBUTES)
  • Composite Vs Simple attributes (which cannot be further divided)
  • Single-Valued Attributes Vs Multi-Valued Attributes (may require a new entity composed of the original multivalued attribute’s components)
  • Derived attributes need not be stored within the database, but may be on some occasions where speed or processing power are limited
  • Relationships - defined with an active or passive verb
  • Connectivity and Cardinality:
    • connectivity in the crow’s foot notation
    • 1:1 / 1:M / M:N is depicted by crow’s foot
    • optional or required depicted by | or O
    • minimum and maximum is represented by ([min], [max]) below the notation on both sides of the relationship
  • Strong Vs Weak Entities:
    • Weak Entities BOTH:
      • are existent dependent
      • have a primary key partially or totally derived from the parent entity
    • A strong identifying relationship indicates that the related entity is weak
  • Relationship Participation:
    • Optional participation - one entity does not require the other, represented by O
    • Mandatory participation - one entity requires the other, represented by |
  • Relationship Degrees:
    • Unary / Recursive Relationships: a unit within a table has a relationship with another unit in the same table. Imagine an employee table where employees may both manage and be managed by, other employees
      • can be 1:1, 1:N or M:N
      • can be shown with crow’s foot notation above itself
    • Binary Relationships: two entities are associated in a relationship - by far the most common type of relationship. Most higher-order relationships will be broken down into binary relationships whenever possible
    • Ternary and Higher-Order Relationships: implies a relationship among 3 or more entities - consider the doctor / patient / drug relationship
  • Associative / Composite Entities: where a M:N relationship exists, this will need to be broken down into 2x 1:N relationships. The entity created in order to achieve this is known as an associative or composite or bridge entity, composed of at least the primary key attributes of each parent entity
  • Compared to an ER diagram, an EER diagram also supports:
    • Entity Supertypes
    • Entity Subtypes
    • Entity Clustering
  • Only use subtypes when:
    • there are different, identifiable kinds or types of environments
    • different instances have one or more attributes that are unique to that instance
  • Specialisation Hierarchy
    • parent vs child subtypes with 1:1 relationships
    • A subtype can only exist within the context of a supertype and every subtype can only have one supertype to which it is related
    • supports attribute inheritance
      • subtype inherits all attributes of supertype
      • all entity subtypes inherit their primary key attribute from their supertype
      • entity subtypes inherit all relationships in which the supertype participate (and any supertype above that supertype)
    • defines supertype attribute: subtype discriminator
      • determines the supertype attribute to which the subtype is related
    • defines disjoint/overlapping constraints and complete/partial constraints
      • disjoint / nonoverlapping subtypes (d): an object cannot belong to more than one subtype at a time
      • overlapping subtypes (o): an object can participate in multiple subtypes simultaneously
        • involves the use of a boolean subtype discriminator
      • Completeness constraint defines weather or not membership in a subtype is mandatory or not:
        • partial completeness: membership not required - single horizontal line
        • total completeness: membership required (at least one) = double horizontal line
  • Specialisation and Generalisation:
    • both are ways in which subtypes and supertypes can be distinguised, defined in opposite methodologies
    • Specialisation: top-down - identify specific subtypes from the higher-level supertype
    • Generalisation: bottom-up - identified by grouping common attributes of subtypes into a supertype
  • Entity Clustering to minimise the number of entities shown in the ERD when things get too busy. This is a virtual entity type used to represent multiple entities for simplification and enhanced readability
  • Primary Keys:
    • arguably the most important aspect of good database design to guarantee entity integrity
    • Natural Keys: a real-world, generally accepted identifier of an object or instance such as a serial number or invoice number. If an object has a unique identifier this it most often its best primary key.
    • The function of a primary key is to identify an entity uniquely, not to describe it.
    • While primary keys are vital to good functioning of a database, they need not always be shown to a user. Always consider the end user point of view.
    • Composite Primary Keys:
      • as identifiers of composite entities in which each primary key combination is allowed only once
      • as identifiers of weak entities in which the weak entity has a strong identifying relationship with the parent entity:
        • a real-world object where existence is dependent on another real-world object
        • a real-world object represented in the database as multiple separate entities
    • Surrogate Keys:
      • a primary key generated by the database designer to simplify identification of entity instances
      • has no meaning in the real-world / user’s environment
      • can be generated by the DBMS to ensure values are always unique
  • Time-Variant Data:
    • data whose values change over time yet a historical record is required - consider the price of an item or a current balance of account
    • May be stored in a different historical data set such as employee history which is different to current employment status
  • Design Traps:
    • occur when a relationship is improperly understood or incompletely identified, leading to an inaccurate representation of the real world relationship
    • Fan Trap:
      • when one entity has two 1:M relationships and thus produces an association among other entities that is not present in the real world. This can occur as a result of > < instead of > >
      • It is important to understand the conduit entity by which the entities are linked, and made sure that this conduit entity is in the middle of the relationship

Week 4: Normalisation

  • The process of removing redundancy from relational databases
    • Redundancy is problematic due to:
      • insertion anomalies
      • update anomalies
      • deletion anomalies
  • 7 Levels of normalisation:
    • Higher forms are usually better but not always; sometimes redundancy is required as less relational join operations are required to respond to queries, resulting in a faster response
    • Occasionally, denormalisation will be required to produce a lower normal form
    • From the data modeler’s point of view, the objective of normalization is to ensure that all tables are at least in third normal form.
    • Zeroth normal form (0NF)
      • Does not meet requirements of 1NF
    • First normal form - 1NF
      • No repeating groups
      • Relations in table format
      • PK identified
      • Conversion to 1NF:
        1. Eliminate Repeating Groups
          1. present the data in tabular format and ensure there are no nulls by making the repeating group attribute have a data value
        1. Identify the Primary Key
          1. may be a composite key
        1. Identify all dependencies
          1. The primary key should identify all else
          1. Determine partial dependencies
          1. Determine transitive dependencies
          1. May be aided by a dependency diagrams

    • Second Normal Form - 2NF
      • meets all requirements of 1NF PLUS has a single attribute primary key
      • If the 1NF does not have a composite key is is already 2NF
      • Conversion to 2NF:
        1. Make New Tables to Eliminate Partial Dependencies
          1. Do this for each of the component keys that has dependants
          1. If A and B are a composite key:
            1. table A
            1. table B
            1. table AB
        1. Reassign Correspinding Dependent Attributes
          1. Take out of AB and put into A or B where possible
          1. Remaining fields remain in AB
          1. Rename all 3 tables where needed
      • note that it is still possible to have transitive dependencies
    • Third Normal Form - 3NF
      • meets all the requirements of 2NF PLUS has no transitive dependencies
      • Conversion to 3NF:
        1. Make New Tables to Eliminate Transitive Dependencies
          1. every transitive determinant will be the primary key in a new table
        1. Reassign Corresponding Dependant Attributes
          1. each dependant attribute can now be removed from the original table
          1. name this table appropriately
  • Functional Dependence:
    • Attribute B is determined by attribute A
    • Partial Dependency :
      • B is determined by A where (A, X) is the primary key
    • Transitive Dependency:
      • B is determined by X which is determined by A(PK)
      • The dependency of one nonprime (non-primary-key) attribute on another
  • Improving on database design :
    • Evaluate PK assignments;
      • Where a title or name exists, have a code linked to that title to prevent misspellings or abbreviations:
        • Such a primary key, developed by the database engineer, is a Surrogate Key
    • Evaluate naming conventions:
      • each attribute related to its table: TAB_ATTRIB
    • Refine Attribute Atomicity:
      • An Atomic Attribute is one that cannot be further subdivided, such as name = firstname + lastname + initial
    • Identify New Attributes
      • such as would be beneficial to the organisation
    • Identify New Relationships
      • add these in without disrupting data integrity or breaking any 3NF rules
    • Refine Primary Keys as Required for Data Granularity
      • data should be granular and clear - is hours worked a daily, weekly or monthly measure?
    • Maintain Historical Accuracy
      • pricing may need to be both current and historical
    • Evaluate Using Derived Attributes
      • only when needed, this may speed up data retrieval
    • Consider Surrogate Keys
      • may be managed by DBMS
        • prevents duplicate entries
        • Must then ensure the original PK it unique
  • The Boyce-Codd Normal Form:
    • Every determinant in the table is a candidate key

Week 5: working with databases on the cloud

  • XML = Extensible Markup Language
  • Modern DBMSs have the ability to store data in geographically distributed nodes
  • Data source and data repository Vs middle layer including database middleware Vs top layer
  • Most DBMS products support other connectivity standards other than their native, most commonly ODBC but also DAO and RDO
  • Open Database Connectivity (ODBC) is Microsoft’s implementation of SQL CLI
    • Data Access Objects (DAO) is an object-orientated API used to access desktop-based databases
    • Remote Data Objects (RDO) is used to access remote database servers, is higher level OO and uses DAO and ODBC
  • Object Linking and Embedding for Database (OLE_DB)
    • database middleware that adds OO functionality for access to nonrelational data
    • Consumers are objects that request data by invoking methods from provider objects
    • Providers are objects that manage the connection with the data source and provide data to the consumer
      • Data providers
      • Service providers (between the data provider and the consumer to add functionality)
  • ADO.NET introduced a simpler object model with Microsoft’s .NET framework support
    • consolidates all data access functionality into one integrated object model
    • is optimised to work in disconnected environments
  • Java Database Connectivity (JDBC)
    • an Application Programming Interface that allows a Java program to interact with a wide range of data sources including relational databases
    • allows programmers to use SQL to manipulate data in a database
  • Web-to-Database Middleware: Server-Side Extensions are programs that interact directly with the server to handle requests
  • Web Server Interfaces:
    • Common Gateway Interface (CGI)
    • Application Programming Interface (API)
  • ActiveX is Microsoft’s alternative to Java for programs that run inside Internet Explorer
  • VBScript is similar
  • Client-side data validation is vital to prevent excessive CPU processing on the server side
  • Web Application Servers are middleware that expand the functionality of web servers by linking them to databases, directories, search engines and the like
  • Web Database Development deals with the process of interfacing databases with the web browser
  • XML:
    • case sensitive
    • tags must be properly formatted
    • must be properly nested
    • can use —> and <— to enter comments
    • is concerned with the description and representation of data, not the way in which it is displayed
    • XHTML expands HTML to include XML features
  • Document Type Definitions (DTD) and XML Schemas
    • file with extension .dtd describes the XML elements
    • the .dtd file must be referenced within the XML
    • XML schemas define data types and validation rules
  • XML presentation:
    • Extensible Style Language Transformation XSLT describes how to extract and process data from XML. Using this you can extract data from an XML and convert it to a text file, HTML or webpage
  • Cloud Computing: a computing model for enabling ubiquitous, convenient, on-demand network access to a shared pool of configurable computer resources that can be rapidly provisioned and released with minimal management effort or service provider interaction

  • ANSI prescribes a standard SQL which is also recognised by the ISO
  • queries are used not only to extract data from the DB but can also be used to perform operations on it
  • Schemas are groups of database objects, often belonging to a single user, to group tables by functions
  • numbers such as postcodes that may begin with a trailing 0 should be defined as char to avoid truncation

  • create table [name] (
    • [column] [data type] [[constraint]],
    • [column] [data type] [[constraint]]);
  • If working with a composite primary key, all of the primary key’s attributes are contained within the parentheses and separated with a comma. If working with a primary key consisting of two attributes, the order of there attributes is important as the indexing starts in order
  • ANSI standards allow for the following constraints:
    • not null
    • unique
    • default
    • check
  • Creating Indexes
    • create [unique]index [indexname] on [tablename](column1, [column2]);
    • useful when you do not want a value to be unique but do want a combination of values to be unique for example, FNAME, LNAME, DOB
    • you can also use this to sort values by type, if this is something that needs to be done regularly
    • to delete:
      • drop index [indexname]
  • Data Manipulation:
    • INSERT:
      • insert into [tablename] values (v1, v2, v3, v4)
      • insert into [tablename](row1, row2, row3) values (v1, v2, v3)
      • To insert one table (or subset) into another
        • insert into [tablename] select [columnlist] from [tablename]
    • COMMIT:
      • saves up to the current point in a similar way to git commit
    • SELECT:
      • select [columnlist (or *)]
      • from [tablename]
      • [where condition];
      • when writing code for a program, it is often best practice to use select followed by all of the variables in comparison to using *. That way, if an table is changed, an error may be thrown and you will not have one value being propagated into the wrong table
    • WILDCARDS:
      • use * for all when used in an attribute or list in a select statement or
      • use % for zero or more characters in a like clause
    • UPDATE:
      • UPDATE tablename
      • SET columnname = expression [, columnname = expression]
      • [WHERE condition(such as line ID)];
      • set-orientated command
      • note that if you do not include the where definition, ALL rows will be altered
    • ROLLBACK:
      • similar to a git undo last commit
      • will NOT delete tables that were added since last commit
        • all data definition commands (create table) are automatically committed to the data dictionary and cannot be rolled back
      • will only undo the result of the insert and update commands
    • DELETE:
      • delete from [tablename]
      • [where contidion(such as line ID];
      • set-orientated command
    • OPERANDS:
      • AND
      • OR
      • NOT
    • SPECIAL OPERATORS:
      • BETWEEN
        • check an inclusive range
        • value between x and y
      • IS NULL
        • must use is null for a null value, otherwise = null looks for a value of ‘null’
      • LIKE
        • check if string matches a pattern
        • WILDCARDS:
          • % - any and all following or preceding chars are eligible
          • _ - any ONE char may be substituted for the underscore
          • SOME RDBMSs also use ? and * in place of % and _
      • IN
        • check if attribute matches a value in a list (a, b, c)
          • the list used can be from another table with
            • (select [element] from [table])
      • EXISTS
        • check if subquery matches any rows
          • if it returns any rows the main query will be run otherwise, it will not
    • ALTER TABLE:
      • alter table tablename
      • {add | alter/modify} (columnname datatype [{add | alter/modify}] columnname datatype]);
      • note that it is not possible to add a column with the not null requirement if the table is already populated as this would break its own rule set
      • Add constraint:
        • alter table tablename
        • add constraint [add constraint];
      • Remove Element:
        • alter table tablename
        • drop {primary key | column columnname | constraint constraintname};
      • Update
        • Update table
        • set column = x
        • where {another}column = y(or range);
      • Create Table from another table:
        • select [column copy as column paste], [repeated]
        • into [new table]
        • from [table to be copied]
        • you would then declare a primary key
          • alter table [new table]
          • add primary key (column);
          • and any foreign keys
            • alter table [new table]
            • add foreign key (column) references [reference];
      • Deleting a table:
        • Drop Table [table]
          • can only be done with the one side of a one to many relationship
    • Order by
      • select column/s
      • from table
      • where condition
      • order by column [asc/desc];
      • cascading order sequence:
        • separate with a comma ie LName, FName, MName;
    • Listing Unique Values
      • to display only the different values of a column:
      • select distinct column
      • from table
    • Aggregate functions
      • Count:
        • will count the number of distinct non-null values
        • select count(column/*)
        • from table;
        • using * will yield the number of total rows from the query, including the rows that contain nulls
      • MAX and MIN:
        • yields only one value
        • If wanting to see the entry for this value:
          • select val1, val2, val3
          • from table
          • where val1 = (select max(val1) from table)
      • SUM:
        • cumulative sum of all rows
      • AVG:
        • average of all rows
    • Grouping Data:
      • can only be used in conjunction with one of the other aggregate functions
      • select column
      • from table
      • where condition
      • group by column
      • having condition
      • order by column [asc/desc]
      • HAVING clause:
        • similar to a where clause in a select statement
        • is applied to the output of a group by statement
    • Joining Tables
      • To join tables, simply list them in the FROM clause of a select statement
      • The WHERE clause becomes the pivot of the link used to match up each line to each other
      • select val1, val2, val3, val4
      • from table1, table2
      • where table1.linkingkey = table2.linkingkey
      • and condition
      • order by val2;
      • if you do not specify the where clause, the result will be a cartesian product of each table ie. 3 rows x 3 rows = 9 possibilities
      • when joining 3 or more tables, you must specify n-1 join conditions in the from clause
        • from a, b, c, d
        • where a=b
        • and b=c
        • and c=d
      • DO NOT join D to A
      • Aliases:
        • during the join, in the from clause:
        • select var1, var2
        • from table a, table b
        • where a.var1 = b.var1
      • Recursive Joins:
        • select var1, var2
        • from tablea a, tablea b
        • where a.var3 = b.var4
