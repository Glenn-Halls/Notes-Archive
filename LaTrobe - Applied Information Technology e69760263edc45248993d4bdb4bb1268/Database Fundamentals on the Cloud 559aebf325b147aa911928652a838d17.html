Database Fundamentals on the Cloud

Week 1: Database Concepts

  • Data (raw facts) Vs Information (processed data)
  • 90% of the data in the world has been created in the last 2 years
  • Database (structured collection of raw facts Vs Database Management System (DBMS) - Software for managing a database including structure and access
  • Database Vs file system
    • File systems may be redundant, not accessible, duplicated (maybe incorrectly), consider concepts of “Islands of information”
    • Database advantages: security, sharing, consistency, productivity, decision making, metadata (data about data)
    • Database disadvantages: complexity, costs, vendor dependence, upgrade/replacement cycles, maintaining currency
  • Consider DBMS as the intermediary between the end user and the database
  • Queries and ad hoc queries
  • Single-user, desktop, multiuser, workgroup (<50), enterprise, centralised, distributed, general-purpose and cloud databases
  • Operational / transactional / production / online transaction processing (OLTP) vs analytical databases (data warehouses and Online analytical processing - OLAP)
  • Concept of Business intelligence, supported in decision making by data warehousing and analytics
  • Unstructured Vs Structured data and Semistructured data
  • Extensible Markup Language (XML) databases to represent and manipulate data elements in a textural format
  • NoSQL = Not only Structured Query Language
  • Database Design - the process that describes the database structure and determines database components
  • Data Processing specialists in the age of file formatting
  • File System Data Processing Problems:
    • Lengthy development times
    • Difficulty getting quick answers
    • Complex system administration
    • Lack of security and limited data sharing
    • Extensive programming required
    • Structural and data dependence
    • Islands of information and data redundancy
  • Logical Vs physical data formats
  • Uncontrolled data redundancy sets the stage for:
    • poor data security
    • data inconsistency
    • data entry errors
    • data integrity problems
    • data anomalies - when not all of the changes required to be made are flowed through to all redundant data repositories
      • update anomalies
      • insertion anomalies
      • deletion anomalies
  • The Database System Environment
    • comprising of hardware, software, people, procedures and data
    • System administrators, database administrators, systems analysts and programmers and end users
  • DBMS functions:
    • data dictionary management
      • abstraction to take care of data component rules and relationships, removing data dependence from the system
    • data storage management
      • including performance tuning
    • data transformation & presentation
      • such as localised date transformations
    • security management
      • including access control
    • multiuser access control
    • backup and recovery management
    • data integrity management
    • database access languages
      • A query language is non-procedural and allows a user to specify what is to be done without having to specify how it is to be done
    • application programming interfaces
    • database communication interfaces
  • Managing the Database System:
    • Disadvantages of a database system:
      • Cost
      • Management complexity
      • Maintaining currency
      • Vendor dependency
      • Frequent upgrade / replacement cycles
  • Data Models:
    • Entities
      • Objects for storing data
      • Data is stored in attributes
    • Relationships
      • Associations between entities
      • Can be 1:1, 1:M, M:M
    • Constraints
      • Rules to ensure data integrity
    • Business Rules:
      • Organisational policies, procedures or principles
      • Can help establish entities, relationships and constraints
    • Historical Structures:
      • File System
      • Hierarchical Model
      • Network Model (multiple parents allowed)
      • Relational Model
        • can solve more advanced informational needs and can now be supported by current generation IT infrastructure
        • A relation is a two dimensional structure with intersecting rows and columns i.e. a table or file
        • An attribute is one column of a relation i.e. field or column
        • A tuple is one line of a relation i.e. a record or row
        • A value is one piece of data in a field i.e. a cell
      • Entity relational model - a graphical representation of a relational model
      • Object-Orientated Model - often used together with OOP where classes replace entities and also includes the features of behaviour and inheritance
      • Big Data Models
        • data sets too large for other models
        • NoSQL
        • Hadoop
        • MapReduce
      • At bare minimum, a data model should contain a description of the data structure that will store the end-user data, a set of enforceable rules to guarantee the integrity of the data and a data manipulation methodology to support real-world data transformations
  • Data Model Basic Building Blocks:
    • Entities:
      • A real-world object - person, place or thing - about which data will be stored
      • Rectangle in an ERD, also known as an entity box
      • Name will be singular and in all-caps
    • Attributes:
      • Characteristics about an entity
    • Relationships:
      • Associations between entities: 1:M, M:N, 1:1
      • Relation is referred to as connectivity, in the form of an active or passive verb
    • Constraints:
      • Restrictions placed on data to ensure data integrity in the form of rules
    • Business Rules:
      • Policies, procedures or principles that define the above building blocks
      • As a general rule, a noun in a business rule will translate to an entity in the model and a verb in a business rule will translate to a relationship among entities
  • Naming conventions:
    • Attributes are often in all-caps
    • It is good practice to prefix the name of an attribute with the name or an abbreviation of the name of the entity in which is occurs for example a customer credit limit = CUS_CREDIT_LIMIT
  • Schema refers to the conceptual organisation of the entire database as viewed by the database administrator
  • Subschema refers to the portion of the database ‘seen’ by the application programs
  • Data manipulation languages (DMLs) and Data Definition Languages (DDLs)
  • The relational model was introduced in 1970
  • Peter Chen first introduced the Entity Relationship Model (ERM and thus ERDs) in 1976
  • Each row in a relational table is known as an entity instance or entity occurrence with a collection of entities known as an entity set
  • The Object Orientated Model - OODM
    • both data and its relationships are contained in a single structure, known as an object
    • Object Orientated Database Management System = OODBMS
    • OODM is said to be a semantic data model
    • OODM objects contain both their attributes and also their functions / operations
      • objects are therefore self-contained and re-usable
      • objects are analogous to an ER model’s entities
      • Objects with similar characteristics are grouped in classes, which may have methods
      • Classes are organised in a class hierarchy
      • Classes have inheritance
    • OODMs are typically depicted using Unified Modelling Language (UML) class diagrams
  • The Extended Relational Database Model - ERDM
    • adds some features of the OODM within the inherently simpler RDM framework
    • includes objects, extensible data types based on class and inheritance
    • is often described as an Object / Relational Database Management System (O/R DBMS)
    • is more focused toward general deployment as opposed to OO’s specificity
    • is easily extended to accommodate new classes and capabilities such as XML
  • Extensible Markup Language (XML) has emerged as the de facto standard within the industry, in order to be able to handle the growing quantity of unstructured data such as diagrams, media, webpages etc.
  • Big data refers to the movement to find new and better ways to manage the large amounts of web and sensor generated data to derive business insights and is defined by the 3 Vs:
    • Volume
    • Velocity
    • Variety
  • Hadoop: java based, open source, high speed, distributed storage and computation. Supported by the Apache Software Foundation.
    • Hadoop Distributed File System (HDFS): storage for big data at high speeds
    • MapReduce: open source API providing fast data analytic services by distributing data processing among thousands of parallel nodes
      • Map: takes a job and splits it into smaller units of work
      • Reduce: collects outputs and recombines into a whole
  • NoSQL: large-scale distributed database system that stores both structured and unstructured data
    • Key-Value data model: ID, Key (attribute), Value (usually a string)
    • very useful for large amounts of sparse data
      • consider a clinic with many patients and many assessments - do not want a blank field for every test that a patient has NOT taken…
    • highly scalable, available and fault tolerant
  • Data abstraction:
    • External Model:
      • End User’s View of the data environment
      • External Model’s View / External Schema
    • Conceptual Model:
      • Global view of the entire database by the entire organisation
      • Conceptual Schema
      • The most widely used being the ER model represented by the ERD
      • independent from both software and hardware
      • Logical Design: creating a conceptual model that could be implemented by any DBMS
    • Internal Model:
      • Maps the conceptual model on the DBMS
      • Internal schema
    • Physical Model:
      • Lowest level of abstraction, describes the way in which data is stored and accessed in the physical world
      • software and hardware dependant
      • When you can change the physical model without affecting the internal model you are said to have physical independence
