Information Systems

  • SDLC = System Development Lifecycle
  • System: A set of related components that produces specific results.
  • Information is data that has been transformed into output that is valuable to users.
  • Information systems contain:
    • 1) Hardware
    • 2) Software
    • 3) Data
    • 4) Processes
    • 5) People
  • Classification of IS (historical)
    • Exectutives (strategic) —> Senior Managers (management) —> Middle Managers (knowledge) —> Workers (operational)
  • Classification of IS (current - functional)
    • Operations support systems
      • Transaction Processing / Process Control / Enterprise Collaboration systems
    • Management support systems
      • Management Information / Decision Support / Executive Information systems
  • Stakeholders:
    • Project Sponsors / Organisational Management / System Users
  • SLDC lays down a clear plan of action to undertake in software development projects to ensure success and to manage the project’s progress
  • The 5 SDLC Stages:
    1. Plan
      1. Why / How / Stakeholders / Cost / Value
      1. Engage with Stakehoders and conduct a SWOT (strengths, weaknesses, opportunities, threats) analysis
    1. Analyse
      1. Study Existing System
      1. Understand Requirements
      1. Create a system proposal
    1. Design
      1. Strategy - insource or outsource? Ability to deliver?
      1. Local design - representation of a system through data flow charts, etc
      1. Physical design including hardware, software, required programming, support etc.
    1. Implementation
      1. Build and Test to ensure the system meets the system goals
      1. Installation and Rollout - clear focus on the stakeholders at this point
    1. Support
      1. Hardware and Software maintenance
      1. Security
      1. User support
  • System Development Methodologies:
    • Traditional Approach
      • Linear, similar to structured programming of the 1960s, structured design of the ‘70s and structured data flow diagrams of the current era
      • Waterfall Development Methodology
        • Pros: strict, great for known factors where iteration is not needed or where users have less experience
        • Cons: strict, inflexible, problematic where desired outcomes change or are unclear
    • Agile Approach
      • More philosophical than methodological
      • An iterative, flexible approach accentuating speed and adaptation
      • Focus on individuals and interactions over process, software over documentation, change over plans, customer collaboration
    • Object Oriented Approach
      • Works symbiotically with OOP
      • Focus on modularity and reusability of each object to form a cohesive whole
      • Can fit in with both traditional and agile approaches

Week 4: Modelling Requirements

  • All systems that interact with the world have three elements in common: input, process and output.
  • Data Flow Diagrams:
    • Symbolism (Gane and Sarson notation):
      • Process - Verb & Noun
        • This represents a single function within a system, containing a unique number in the upper section and a description in the lower section
      • External Entity
        • These either feed data to the system or receive data from the system.
      • Data Store
        • The left section contains the letter D followed by a unique identifier with the right side containing a noun of the data store
    • As a general rule, DFDs should have three to seven processes at each level of depth
    • Context Diagrams
      • The highest level DFD is a context diagram, representing all external entities and the system as a single process bearing the number 0 and a descriptor ending in the word “system”
      • Breaking down further to more minute detail is known as Decomposition or levelling. Consider both numbering (1.2.2.3) and balancing (inflow and outflow remains consistent throughout levels) when doing so. This process may be considered complete when:
        • Each process has been narrowed to a single task
        • Each data store stores data on a single entity
        • Data flows no longer need to be split to show different data handling
    • Functional Decomposition Diagram
      • Top-Down tool to show the structure of an organisation or system
    • Business Process Modelling
      • Graphically displays one or more business processes
    • Unified Modelling Language
      • Widely used to visualise and document software systems design
    • Level 0 DFD
      • Shows the core internal processes of a system including data stores and data flows

Week 6: Process Modelling

  • Process description techniques
    • Structured English
      • if / elseif / endif
      • Plain English —> Structured English —> Pseudocode —> Programs
      • All logic is stated in terms of sequential statements, iteration statements, decision structures (if/else) and case structures (if/elseif/else)
      • Indented as per coding
      • capatilise keywords IF, ELSE, THEN ENDIF
      • comment using *
      • be cautious and consider using ( ) when using and / or
    • Decision Table
      • Matrix format ensures all possible combinations have been considered
      • Decision Rules / Conditions / Actions / Condition Entries (Y/N) / Action Entries (Y/N)
      • Condition Stub —> Actions Stub —> Calculate Columns / Rules —> Populate —> Simplify by combining irrelevant information with a “-” for N
    • Decision Tree
      • A graphical representation of a business process and its programming logic.
      • Nodes = conditions or actions
        • Condition Node = circle. Branches from this are the possible outcomes of the condition
        • Action Node = Square. This is the action to be performed resulting from the conditions being met.
      • Branches = outcomes
      • Identify all conditions —> Identify all actions —> Build tree from left to right or top to bottom —> At each node, make sure all conditions are listed

Week 7: Data Modelling

  • Entity Relationship Diagrams
    • Yield a conceptual schema of the database for an information system. Contains entities, their attributes and the relationships between them
  • Entities: person / place / object / event / concept. Must be a noun in singular form with an adjective only when needed
  • Consider that often, Data Stores from a DFD may be the entities for an ERD
    • Crow’s foot notation
    • Types - entities that share characteristics
    • Instances - individual entities
    • Entity Sets - share a condition / value
    • Independent Vs Dependent
    • Attributes
    • Tables (relations), fields (columns) and records (rows)
  • Attributes
    • Single-valued attributes
    • Multi-valued attributes {contain in curly braces}
    • Composite attributes (..) is used as a suffix
    • Derived attributer [contain in square brackets]
    • Complex attributes are both multi-valued and composite - in this case use both annotations ie. {Address}(..)
  • Primary Keys
    • Used to uniquely identify an individual entity
    • Selected from candidate keys OR composite keys
    • Must always exist (cannot be NULL), must never change and should be efficient
  • Relationships between entities
    • always related with a line and labelled with an active verb
  • Foreign Keys
    • Establish relationships between entities within tables
    • Referential Integrity - for every foreign key in use there must be a valid primary key in the parent table
  • Entity Relationships
    • Binary
      • Entity instance in one entity type relates to entity instance/s in another entity type
    • Unary
      • Entity instance in one entity type relates to another entity instance of the same entity type
    • Ternary
      • This relationship contains three participating entities
        • Doctor treats patient
        • Doctor prescribes medicine
        • Patient takes medicine
    • Cardinality
      • Specifies how many instances of an entity relate to how many instances of another entity
    • Modality
      • Defines weather or not participation in a relationship within an entity is optional or mandatory
      • Mandatory = |
      • Optional = O
    • Associative Entities
      • Can be used to break up many - many relationships as these otherwise cannot be represented in a relational database. This is achieved by inserting two separate one-many relationships in between the many-many relationship.
      • Achieved by the use of composite keys
      • In an ERD, every many-many relationship must be broken down into 2x one-many relationships

Week 8: Data Modelling 2

  • Data normalisation is a bottom-up approach as opposed to a top-down approach such as with an ERD or a DFD
  • Data normalisation and ERDs can both be used to create a database table and as such, one can be used to validate the other
  • Normalisation is the systematic process of grouping data into separate tables to minimise data redundancy and prevent anomalies
  • Levels of database normalisation:
    • 1NF - enforce the rules
      • Rule 1: each cell should contain a single atomic value (assuming a primary key)
      • Rule 2: values in each column should be of the same type (domain)
      • Rule 3: all columns should have a unique name
      • Rule 4: records should be able to be reordered without consequence
    • 2NF - remove partial dependencies - dependence only on primary keys
      • Dependency diagrams
      • To resolve this, attributes with partial dependencies must be separated into different tables
    • 3NF - remove transitive dependencies - dependence only on primary keys
      • a non-primary-key attribute cannot identify another non-primary key attribute
      • C4-C5 must be separated, with C4 becoming a primary key
    • 4-6NF
      • Not always required but should, if all completed, solve all potential data modification anomalies
  • Data Dictionaries
    • Data Elements (attributes)
      • Data Element Name
      • Alias (optional)
      • Type (alphabetical, numeric, alpha-numeric, text)
      • Length
      • Default Value (can be empty / null)
      • Acceptable Values (may be an example)
      • Source
      • Description and comments
    • Data Records
      • Name
      • Description
      • Aliases
      • Attributes - list of all data elements that make up the data record

Week 9 - User Interface Design

  • Human-Computer-Interaction (HCI) is a discipline that focusses on design of information systems and computing devices for human use, considering the psychology behind the user and the mechanisms behind the code.
  • UI design falls directly in to the Design phase of the SDLC. This is one of the key factors in user acceptance of the system.
  • Documentation of the UI are codified in design documents called the interface design.
  • Consider the difference between UI and system interface
  • 5 Design Principles of a good UI:
    • Appearance:
      • Layout and Design - color, screen density, fonts, easy to use and learn, intuitive design, clarity, hide irrelevant aspects (ie only show submit button when required inputs have been provided), effective use of graphical elements
    • Consistency:
      • Predictability, help (easily accessable, context-sensitive, non-distractive, help desk?) and feedback (confirmations, corrections, progress, tooltips
    • Efficiency:
      • Enhanced user productivity - ease of use and speed of use (shortcuts, prefilling data inputs, autocomplete), reduced input volume.
    • Security & Error Mitigation:
      • User validation (data format awareness - consider field population, input masking, input validation prior to processing
      • Error handling including error messages
      • Reversal of action, action confirmations, breaking up tasks into sections, reviews and summaries of actions / inputs
    • Input / Output Security and Control:
      • This is the fundamental aspect of data security
      • Limit the number of physical outputs like printouts
      • Ensure sensitive data is only received by the intended recipients
      • Securely store physical records
      • Destroy unused output
      • Output restrictions such as restricting access to the organisational intranet and disallowing removable media
      • Input control at every stage of input design
      • Traceability
      • Data security policies including backup, theft protection, access and restriction
  • Output design:
    • Printed outputs
    • Turnaround Documents
    • Detailed reports - minimally filtered
    • Exception report - only records that satisfy a given condition
    • Summary Report - broad overview to identify trends
    • Design Principles:
      • Must have
        • Report headers and footers
        • Page headers and footers
      • Repeating fields
      • Consistent design
      • Consider charts to represent information visually

Week 10: System Implementation

  • One of the most expensive phases of the SLDC in terms of both money and time
  • System Development:
    • Review the system design
    • Application development tasks
    • System development tools
      • Structured approach relies on DFDs and structure charts
      • OO approach relies on the different UML diagrams with agile iterative models
      • Project Management
    • Development Approaches:
      • Input, Process, Output (IPO):
        • Develop programming modules that deal with data entry into the system, then modules that process the data, then modules that present system output.
        • User interface is developed 1st, giving longer to tweak this element and the opportunity for input testing the rest of the system.
        • Outputs are developed last, making it harder to verify that the system is working as intended
      • Top Down and Bottom-Up:
        • Start with a structured model of the program and then build either top-down or bottom up:
          • Top down starts with a scaffold of the system with dummy versions and then builds out from there
          • Bottom-Up starts by building bottom-level modules, then combining them to create the process above until the whole structure is built.
  • System Testing:
    • Development is not complete until this process, along with documentation, has been completed
    • Test Planning defines a series of tests to be performed and specifies each person’s role during testing.
    • Types of software testing:
      • Unit Testing:
        • Performed on a single unit, module, or functionality in the program
        • Black-Box vs White-Box testing
      • Integration Tests:
        • Ensures that modules or programs that are designed to work together do so without errors
        • Can only occur after each module has passed its own unit tests. There are four approaches to this:
          • User interface testing
          • User scenario testing
          • Data flow testing
          • System interface testing
      • System Tests are similar to integration tests but are broadened out to the entire system
      • Acceptance Tests by users under guidance of the development team: alpha and beta testing
  • Documentation:
    • Program Documentation for development and maintenance of a system’s software
    • System Documentation for processes and functions that the system performs and how they are implemented including system modelling, DFDs, data dictionaries
    • Operations Documentation aimed at the IT group where the system involves centralised servers or mainframes
    • User Documentation for the end user, including:
      • System overview
      • Menu / UI overview
      • Example reports, etc
      • Security / audit trail
      • Input / output / processing requirements
      • Procedures for reporting problems
      • Examples of problems or errors
      • FAQs
      • Instructions on updating / how to get help
  • Training and support:
    • Who:
      • Primary users (directly interface) and Secondary users (interface with the outputs of the system), with focus proportional to how much this will change their work
      • Who should deliver this training? Vendors, system analysts, external trainers, in-house trainers and even other system users
    • What:
      • The tasks that they will regularly perform
    • How:
      • Classroom, one-on-one, computer-based
  • System Installation: often a combination of:
    • Direct Installation:
      • The old system is switched off and the new system is switched on
      • Risky but cheap
    • Parallel Installation:
      • The new system is run alongside the old system until technicians and users are ready to switch off the old system
      • safe but expensive and redundant
    • Single-location Installation:
      • Installation of the new system in one or more locations prior to switching over.
      • In these locations either a direct or parallel installation may be used
      • This has similar downsides to parallel installation
    • Phased (Staged) Installation:
      • Progressive installation of components of a system over time, to the whole organisation
      • Often requires bridging software to ensure coexistence of the old and new systems
  • Data Conversion:
    • Migration of data from the old system to the new
    • Existing data must be error free and may even need reformatting
    • Electronic Records:
      • Reusing existing data
      • Modifying existing data
      • Migrating data
    • Security and control
      • Identify and fix data issues early
      • Auditing is essential
  • System Implementation short list)
    • Application Development
      • OO Development
        • Objects grouped into classes
      • Structured
        • Top-down / modular, assigning modules to individuals / teams
      • Agile methods
        • Constant communication with end users
        • Small teams, intense communication
    • Testing
    • Documentation
    • Training
    • Data Conversion
    • System Changeover
    • Post-implementation evaluation

Week 12: Other Development Methodologies

  • Waterfall Software Development Methodology
  • Agile Software Development
    • Based on the values and principles of the “Agile Manifesto”
      • Individuals and interactions over processes and tools
      • Working software over comprehensive documentation
      • Customer collaboration over contract negotiation
      • Responding to change over following a plan
    • Known for its flexibility - tasks are completed concurrently with adapting and adjusting on the fly
    • Requires stakeholder and developer cooperation
    • Broken down into phases known as “sprints” of: Plan —> design —> develop —> test —> deploy —>review —> Plan (and communicate) into next sprint with final stage finishing with “Launch”
    • Agile Frameworks:
      • Scrum
        • One of the most commonly used agile frameworks
        • An iterative model using fixed-length sprints lasting 1-2 weeks
        • At the end of each sprint stakeholders and team members plan the next steps
        • Divided into:
          • Sprint Planning
          • Daily Stand-Up
          • Sprint Demo
          • Sprint Retrospective
      • Extreme Project Management / Megaproject (XPM)
        • Value Experience
        • Embrace Change
        • Sprint
        • Extreme Programming
    • Waterfall Model:
      • Requirementr
      • Design
      • Implementation
      • Verification
      • Maintenance
